# Jhin MVP 文档 - 阶段 0：项目骨架与基础接入

## 1. 阶段目标

本阶段只处理 Mod 的基础接入问题，目标是先让工程可以被游戏正确加载，并让烬角色可以正常进入战斗。

本阶段不实现完整玩法机制，不实现子弹、华彩、标记、莲华陷阱、谢幕等正式战斗系统，只完成后续开发所需的最小骨架。

## 2. 本阶段交付内容

必须交付以下内容：

1. 建立基础目录结构。
2. 创建 `JhinCharacter`。
3. 注册角色颜色、初始卡组、初始遗物。
4. 加入基础本地化文件。
5. 创建抽象基类：`AbstractJhinCard`、`AbstractShootCard`。

## 3. 目录结构要求

需要建立或补齐以下目录：

- `characters/`
- `cards/`
- `powers/`
- `relics/`
- `actions/`
- `ui/`
- `localization/`

结合当前项目，建议这些目录落在 `src/` 下，由代码目录与资源目录分离管理。

建议结构如下：

```text
src/
  actions/
  cards/
  characters/
  powers/
  relics/
  ui/

JhinMod/
  localization/
```

说明：

- `src/characters/` 用于角色类与角色注册相关逻辑。
- `src/cards/` 用于卡牌基类与后续具体卡牌实现。
- `src/powers/`、`src/relics/`、`src/actions/`、`src/ui/` 先建立骨架目录，允许当前阶段仅放占位或空实现。
- 本地化资源优先放在 `JhinMod/localization/`，以便与资源路径保持一致。

## 4. 实现范围

## 4.1 角色接入

需要完成：

- 创建 `JhinCharacter`。
- 将烬注册为可选角色。
- 配置角色基础信息。
- 配置角色初始卡组。
- 配置角色初始遗物。

角色基础数值要求：

- 初始生命：70
- 初始金币：99
- 初始遗物：`低语`

初始卡组要求：

- `普通射击 x4`
- `优雅走位 x4`
- `致命华彩 x1`
- `装填 x1`

本阶段允许这些卡牌先使用最小可运行实现，只要能被加入初始卡组并正常进入战斗即可。

## 4.2 角色颜色与基础资源注册

需要完成：

- 注册烬角色颜色。
- 接入角色卡牌边框、能量球或必要颜色资源引用。
- 保证卡牌创建与渲染链路不因缺少角色颜色而报错。

资源要求：

- 未明确提供正式资源时，角色相关图使用 `JhinMod\Images\role_placeholder.png`。
- 卡牌图使用 `JhinMod\Images\card_placeholder.png`。

本阶段重点是“能加载、能显示、不卡死”，不是最终视觉效果。

## 4.3 初始遗物接入

需要完成：

- 创建 `低语` 的最小可运行版本。
- 将其注册为烬的初始遗物。
- 保证开局时角色持有该遗物。

本阶段要求：

- 遗物至少能够被正确创建、显示、加入角色遗物栏。
- 如果完整效果尚未实现，可先保留最小描述和基础挂接，但不能影响开局与战斗稳定性。

## 4.4 基础本地化文件

需要完成基础本地化文件的创建与加载，至少包括：

- 角色文本
- 卡牌文本
- 遗物文本
- 关键词文本

建议最少文件：

- `JhinMod/localization/zhs/JhinMod-Card-Strings.json`
- `JhinMod/localization/zhs/JhinMod-Relic-Strings.json`
- `JhinMod/localization/zhs/JhinMod-Character-Strings.json`
- `JhinMod/localization/zhs/JhinMod-Keyword-Strings.json`

最低要求：

- `JhinCharacter`
- `低语`
- `普通射击`
- `优雅走位`
- `致命华彩`
- `装填`

都必须拥有可显示的名称和描述，避免游戏内出现空文本或 key 泄漏。

## 4.5 抽象卡牌基类

需要创建：

- `AbstractJhinCard`
- `AbstractShootCard`

### AbstractJhinCard

职责：

- 作为所有烬卡牌的公共父类。
- 封装角色颜色、资源路径、基础构造逻辑。
- 为后续统一接入关键字、升级文本、卡图路径提供扩展点。

本阶段最低要求：

- 能被普通烬卡牌继承。
- 不影响卡牌正常创建、复制、升级和加入牌组。

### AbstractShootCard

职责：

- 作为所有带“射击”概念卡牌的抽象父类。
- 先定义射击牌这一分类，为后续接入子弹消耗和华彩判定预留接口。

本阶段最低要求：

- 提供清晰的扩展点，例如 `isShootCard`、`OnShoot` 或类似标记能力。
- 当前阶段可以不实现真实子弹逻辑，但类结构必须可扩展。

## 5. 非目标

以下内容不属于本阶段：

- 子弹系统正式实现
- 华彩判定与触发
- 标记系统
- 莲华陷阱系统
- 谢幕卡条件系统
- 完整 88 张卡池
- 平衡性调整
- 单元测试

## 6. 推荐最小实现策略

为了尽快达成“能进游戏、能进战斗、不崩溃”，建议本阶段采用最小实现：

1. 先完成角色注册与选角入口。
2. 用最小可运行版本实现 4 张初始卡。
3. 用最小可运行版本实现初始遗物 `低语`。
4. 完成本地化最小集合，保证界面文本可读。
5. 先把目录、基类和注册链路搭好，再进入后续核心机制阶段。

这里的关键不是数值和玩法完整，而是工程入口稳定。

## 7. 验收标准

本阶段完成后，必须满足以下验收条件：

1. 能选择烬进入游戏。
2. 初始生命正确，为 70。
3. 初始金币正确，为 99。
4. 初始遗物正确，为 `低语`。
5. 初始卡组正确：`普通射击 x4`、`优雅走位 x4`、`致命华彩 x1`、`装填 x1`。
6. 进入战斗后不会因为角色、初始卡组、遗物、本地化或资源缺失而崩溃。

## 8. 完成定义

当以下条件全部满足时，可判定“阶段 0 完成”：

- 项目目录骨架已建立。
- 烬角色已注册并可选。
- 初始卡组与初始遗物已接入。
- 基础本地化文件已可用。
- `AbstractJhinCard` 与 `AbstractShootCard` 已建立。
- 至少可以正常开始一场战斗且无崩溃。

## 9. 下一阶段接口预留

为了减少返工，本阶段代码结构应提前为下一阶段预留以下扩展点：

- 角色战斗状态字段挂载位置
- 射击牌统一结算入口
- 遗物监听华彩事件的入口
- 关键词与 tooltip 扩展入口
- 子弹 UI 挂载位置

这部分只要求预留结构，不要求在本阶段完成具体实现。
