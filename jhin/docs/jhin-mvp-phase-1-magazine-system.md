# Jhin MVP 文档 - 阶段 1：弹匣系统 MVP

## 1. 阶段目标

本阶段目标是实现烬最核心的“4 发子弹”节奏，让角色在战斗中具备基础的弹匣资源管理能力。

本阶段只处理弹匣资源本身与射击牌的基础联动，不扩展到完整华彩、标记、莲华陷阱、谢幕等高级玩法。

## 2. 本阶段交付内容

必须交付以下内容：

1. 实现子弹相关字段。
2. 实现战斗开始时获得 4 / 4 发子弹。
3. 实现射击牌消耗 1 发子弹。
4. 实现子弹为 0 时普通射击牌不可使用。
5. 实现回合开始时仅在空弹匣情况下自动装填。
6. 建立弹匣系统的基础 Action / Power / 卡牌支撑类。

## 3. 需要实现的字段

本阶段至少需要维护以下字段：

- `bullets`
- `maxBullets`
- `flourishCountThisTurn`
- `flourishCountThisCombat`
- `usedShootThisTurn`

字段要求：

- `bullets` 表示当前子弹数。
- `maxBullets` 默认值为 4，表示当前战斗的子弹上限。
- `flourishCountThisTurn`、`flourishCountThisCombat` 先完成字段接入与基础重置逻辑，为后续华彩系统做准备。
- `usedShootThisTurn` 用于记录本回合是否使用过射击牌，为后续卡牌和能力联动预留。

## 4. 功能需求

## 4.1 战斗开始初始化

需要完成：

- 进入战斗时初始化子弹系统。
- 设置 `maxBullets = 4`。
- 设置 `bullets = 4`。
- 初始化本回合与本场战斗计数字段。

最低要求：

- 每场战斗开始时都能稳定回到 4 / 4 子弹状态。
- 不受上一场战斗残留状态影响。

## 4.2 射击牌消耗子弹

需要完成：

- 所有继承 `AbstractShootCard` 的普通射击牌在使用时消耗 1 发子弹。
- 每次成功打出射击牌后，`bullets` 减 1。
- `usedShootThisTurn` 在本回合第一次使用射击牌后更新为正确状态。

最低要求：

- 单段射击牌结算稳定。
- 当前阶段先确保基础单次消耗正确，多段射击特例可在后续阶段补强，但不能破坏当前结构。

## 4.3 空弹限制

需要完成：

- 当 `bullets == 0` 时，普通射击牌不可使用。
- 不可使用状态应在卡牌可用性判断中生效，而不是等到结算时才报错。
- 必须向玩家返回明确的不可用原因。

最低要求：

- 空弹时射击牌不会被打出。
- 不会出现子弹为负数的情况。
- 不会因空弹使用射击牌导致战斗异常或崩溃。

## 4.4 回合开始自动装填

需要完成：

- 在回合开始时检查当前子弹数。
- 如果 `bullets == 0`，自动装填至 `maxBullets`。
- 如果 `bullets != 0`，保持当前子弹数不变。
- 回合开始时同步重置 `flourishCountThisTurn` 和 `usedShootThisTurn`。

最低要求：

- 自动装填只发生在空弹匣状态。
- 剩余 1、2、3 发子弹时不会被强制补满。
- 不会出现“每回合无脑回满弹匣”的错误行为。

## 5. 建议先做的类

本阶段建议优先实现以下类：

- `BulletPower`
- `ShootAction`
- `ReloadAction`
- `AbstractShootCard`

## 5.1 BulletPower

职责建议：

- 作为子弹系统的主要状态承载体，或作为角色战斗状态的统一显示入口。
- 负责对外展示当前子弹数和最大子弹数。
- 为后续 UI、华彩、装填监听提供统一状态来源。

本阶段最低要求：

- 能正确反映当前 `bullets / maxBullets`。
- 不要求完整视觉表现，但需要保证状态同步可靠。

## 5.2 ShootAction

职责建议：

- 统一处理射击牌的子弹消耗逻辑。
- 在射击行为结算前进行子弹合法性检查。
- 成功射击后更新角色子弹状态。

本阶段最低要求：

- 至少能处理“消耗 1 发子弹”的基础流程。
- 应避免把消耗子弹逻辑分散写进每一张射击牌里。

## 5.3 ReloadAction

职责建议：

- 统一处理装填逻辑。
- 在回合开始空弹时调用。
- 后续也可复用于卡牌、遗物、药水带来的主动装填效果。

本阶段最低要求：

- 支持将当前子弹恢复到 `maxBullets`。
- 保证与战斗开始初始化逻辑兼容。

## 5.4 AbstractShootCard

职责建议：

- 作为所有“射击牌”的公共入口。
- 统一定义射击牌可用性判定。
- 为后续扩展华彩、免费射击、多段射击预留接口。

本阶段最低要求：

- 能识别自己属于射击牌。
- 在 `bullets == 0` 时正确返回不可用。
- 打出时走统一的射击结算路径。

## 6. 非目标

以下内容不属于本阶段：

- 华彩正式触发逻辑
- 华彩额外效果与华彩反馈
- 标记系统
- 莲华陷阱系统
- 谢幕系统
- 完整射击特例支持，例如“不消耗子弹”“下一张射击不消耗子弹”“强制华彩”等
- 单元测试

## 7. 推荐最小实现策略

建议按以下顺序实现：

1. 先确定子弹字段挂载位置，统一状态来源。
2. 完成战斗开始初始化与回合开始自动装填。
3. 在 `AbstractShootCard` 中接入空弹不可用判断。
4. 用 `ShootAction` 落实射击时消耗 1 发子弹。
5. 用 `ReloadAction` 统一装填行为，避免后续重复实现。

重点是先把“4 发子弹节奏”跑通，而不是提前实现复杂变体。

## 8. 验收标准

本阶段完成后，必须满足以下验收条件：

1. 射击牌每次使用减少 1 发子弹。
2. 0 发子弹时不能使用射击牌。
3. 回合开始时只有空弹匣才自动装填。
4. 不会出现每回合无脑回满弹匣的问题。

建议补充手动验证场景：

1. 连续打出 4 张射击牌后，子弹应从 4 依次降到 0。
2. 第 5 张普通射击牌在同回合应不可打出。
3. 若回合结束时仍剩 1 到 3 发子弹，下回合开始后应保持原数值。
4. 若回合结束时子弹为 0，下回合开始后应恢复到 4。

## 9. 完成定义

当以下条件全部满足时，可判定“阶段 1 完成”：

- 子弹字段已稳定接入角色战斗状态。
- 射击牌与子弹消耗已联动。
- 空弹限制已生效。
- 空弹自动装填已生效。
- 不存在“每回合自动补满弹匣”的错误实现。
- 进入战斗、打牌、回合切换过程中无崩溃。

## 10. 下一阶段接口预留

为了减少后续返工，本阶段应预留以下扩展点：

- 华彩触发判定入口
- 本回合与本场战斗华彩计数的更新点
- 射击牌特殊消耗规则扩展点
- 子弹 UI 展示接口
- 遗物、能力牌监听射击或装填事件的入口

这部分只要求结构可扩展，不要求本阶段完成完整机制。
