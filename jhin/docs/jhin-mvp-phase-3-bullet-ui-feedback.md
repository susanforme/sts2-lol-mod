# Jhin MVP 文档 - 阶段 3：弹匣 UI 与战斗反馈

## 1. 阶段目标

本阶段目标是让玩家能够直接看见当前子弹状态，并在打牌前清楚判断下一张射击是否会触发华彩。

本阶段重点是“可视化”和“交互反馈”，解决机制已经存在但玩家看不见、看不懂、看不准的问题。

## 2. 本阶段交付内容

必须交付以下内容：

1. 实现 `BulletCounterUI`。
2. 在角色血条附近显示 4 个子弹图标。
3. 用 `●` 表示有弹，用 `○` 表示空弹。
4. 剩 1 发子弹时高亮最后一颗子弹。
5. 当前射击牌会触发华彩时，通过卡牌边框或 Tooltip 给出提示。
6. 子弹为 0 时，射击牌置灰并提示无法使用。

## 3. 功能需求

## 3.1 子弹计数 UI

需要完成：

- 在角色血条附近显示子弹状态。
- UI 固定显示 4 个位置，对应当前最大基础弹匣节奏。
- 每个位置根据当前子弹数显示“有弹”或“空弹”。

显示规则：

- 有弹：`●`
- 空弹：`○`

示例：

- 4 发子弹：`● ● ● ●`
- 2 发子弹：`○ ○ ● ●`
- 1 发子弹：`○ ○ ○ ●`
- 0 发子弹：`○ ○ ○ ○`

最低要求：

- UI 能随着子弹变化实时刷新。
- 战斗开始、射击消耗、回合开始自动装填后显示都必须正确。

## 3.2 最后一发高亮

需要完成：

- 当当前子弹数为 1 时，对最后一颗子弹进行视觉高亮。
- 高亮的目的不是美术增强，而是明确提示“下一张有效射击将触发华彩”。

最低要求：

- 玩家无需读取说明文本，仅看子弹 UI 就能知道当前处于“第四枪前一发”状态。
- 高亮状态在子弹数变化后立即更新。

## 3.3 华彩预告提示

需要完成：

- 当当前子弹为 1，且一张射击牌如果打出将真实消耗最后一发子弹时，对该牌显示华彩提示。
- 提示方式可采用卡牌边框高亮、卡牌标题高亮或 Tooltip 提示。
- 若射击牌由于特殊规则不会消耗子弹，则默认不显示华彩提示。

最低要求：

- “下一张会触发华彩的射击牌”必须有明显提示。
- 普通非射击牌不能错误显示华彩提示。
- 不消耗子弹的射击牌不能错误显示华彩提示，除非未来专门扩展特殊规则。

## 3.4 空弹状态下的卡牌反馈

需要完成：

- 当 `bullets == 0` 时，普通射击牌在手牌中显示为不可使用状态。
- 卡牌应置灰或使用当前项目的标准不可用视觉反馈。
- 同时应提供明确的 Tooltip 或说明文字，例如“没有子弹，无法使用”。

最低要求：

- 玩家在出牌前就能知道这张牌当前不能用。
- 不允许仅在点击或打出失败后才给反馈。
- 空弹提示应稳定、明确、可重复触发。

## 4. 建议实现的类与入口

本阶段建议优先完善以下类或接口：

- `BulletCounterUI`
- `BulletPower`
- `AbstractShootCard`
- 角色或战斗 HUD 挂载入口
- 卡牌可用性与 Tooltip 提示入口

## 4.1 BulletCounterUI

职责建议：

- 负责在战斗界面中渲染子弹数量。
- 根据当前 `bullets / maxBullets` 刷新图标状态。
- 在仅剩 1 发时提供特殊高亮表现。

本阶段最低要求：

- 可以稳定读取当前角色的子弹状态。
- 可以在战斗过程中实时更新显示。
- 即使使用的是最小占位 UI，也必须保证信息表达清楚。

## 4.2 BulletPower

职责建议：

- 作为 `BulletCounterUI` 的状态来源之一。
- 为 UI 提供当前子弹值、最大子弹值与是否处于华彩预备状态的信息。

本阶段最低要求：

- UI 层不应自己维护一份独立子弹状态。
- 状态源必须唯一，避免显示与实际逻辑不一致。

## 4.3 AbstractShootCard

职责建议：

- 统一处理射击牌在当前状态下的可用性判断。
- 提供华彩预告条件判断接口。
- 提供空弹时的不可用提示文本。

本阶段最低要求：

- UI 与可用性判断共享同一套逻辑来源。
- 不允许 UI 显示“可华彩”，但实际打出后不触发的明显错误。

## 5. 视觉与资源要求

本阶段以“表达清楚”为第一优先级。

资源要求：

- 若未提供正式 UI 素材，可先使用最小图形、字符或占位表现。
- 若需要角色相关资源，使用 `JhinMod\Images\role_placeholder.png`。
- 若需要卡牌资源，使用 `JhinMod\Images\card_placeholder.png`。

实现要求：

- 不要求本阶段做复杂特效。
- 不要求最终美术风格定稿。
- 但必须保证玩家能明确读懂状态。

## 6. 非目标

以下内容不属于本阶段：

- 新的战斗机制实现
- 标记系统
- 莲花陷阱系统
- 谢幕系统
- 完整华彩演出特效、音效和屏幕暗化表现
- 最终品质美术资源
- 单元测试

## 7. 推荐最小实现策略

建议按以下顺序实现：

1. 先完成 `BulletCounterUI` 的基础显示。
2. 接入战斗状态刷新，保证子弹变化时 UI 正确更新。
3. 在只剩 1 发时增加最后一发高亮。
4. 在射击牌可用性判断里统一接入空弹提示。
5. 最后补华彩预告提示，确保“看见的提示”和“真实的触发条件”一致。

重点是先把状态表达准确，再考虑视觉优化。

## 8. 验收标准

本阶段完成后，必须满足以下验收条件：

1. 玩家能一眼看出当前子弹数。
2. 玩家能明确知道下一张射击是否会触发华彩。
3. 空弹状态下的射击牌反馈清晰。

建议补充手动验证场景：

1. 战斗开始时应显示 `● ● ● ●`。
2. 每打出 1 张普通射击牌后，UI 子弹数应正确减少。
3. 当子弹剩 1 发时，最后一颗子弹应有明显高亮。
4. 当子弹剩 1 发时，普通射击牌应显示“将触发华彩”的提示。
5. 当子弹为 0 时，普通射击牌应置灰，并提示“没有子弹，无法使用”。
6. 回合开始自动装填后，UI 应立即恢复为 `● ● ● ●`。

## 9. 完成定义

当以下条件全部满足时，可判定“阶段 3 完成”：

- `BulletCounterUI` 已可用。
- 子弹 UI 与实际战斗状态一致。
- 剩 1 发时存在清晰的华彩预备提示。
- 空弹时射击牌不可用反馈完整。
- 玩家无需试错即可理解当前能否射击、是否会华彩。

## 10. 下一阶段接口预留

为了减少后续返工，本阶段应预留以下扩展点：

- 华彩触发后的特效与音效入口
- 谢幕类卡牌专属演出提示入口
- 标记与莲花陷阱的目标状态提示入口
- Boss 遗物改变最大弹匣后 UI 动态扩展入口

这部分只要求结构可扩展，不要求本阶段实现完整内容。
